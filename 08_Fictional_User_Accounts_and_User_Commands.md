# Fictional User Accounts and User Commands

## 1) Executive Summary
This deep-dive artifact formalizes the existing privacy-safe audit fixture package used for QA, demos, and audit evidence. The canonical fixture source currently contains **17 fictional user accounts** and **45 realistic user commands**, all using synthetic identifiers only.

The package covers five role clusters:
- `guest`
- `learner`
- `instructor_creator`
- `admin_support`
- `manager_guardian`

The command library covers all eight required journey categories:
- onboarding and authentication
- course discovery, enrollment, and purchase
- learning experience
- content creation and moderation
- support operations
- data rights and privacy
- notifications and preferences
- error and recovery paths

Edge-case coverage is intentionally built in, including accessibility-critical learners, low-bandwidth use, failed renewals, suspicious-login locks, duplicate-email recovery, guardian consent for minors, privacy-rights requests, unsubscribed marketing, moderation review, and enterprise access governance.

This document is synchronized to the typed fixture source in `apps/frontend/src/lib/audit-user-fixtures.ts` and its validation coverage in `apps/frontend/tests/audit-user-fixtures.integration.test.ts`. It is generic and reusable by design and does not claim any live-system observation or real-user evidence.

## 2) Scope & Method
**Scope**
- Formalize the existing fictional account and user-command fixture set into an audit-ready written artifact.
- Preserve the typed fixture library as the source of truth rather than creating a second disconnected dataset.
- Cover realistic role behavior, edge cases, workflow completeness, privacy safeguards, and testability.

**Method**
- Reviewed the typed audit fixture source in `apps/frontend/src/lib/audit-user-fixtures.ts`.
- Reconciled the written summary with the actual exported fixture counts and coverage.
- Preserved only synthetic identifiers: fictional names, `ql.test` email addresses, `example.*`-safe patterns when applicable, and `+1-555-010-xxxx` phone formatting.
- Mapped commands to role clusters and journey categories so each command remains traceable to audit scope.
- Kept the artifact neutral and reusable for QA, demos, and audit evidence without asserting observed production behavior.

**Traceability model**
- Account identifiers use `AC-xx`.
- Command identifiers use role-prefixed ids: `L-xx`, `I-xx`, `A-xx`, `M-xx`.
- Each command references one or more account ids so evidence can be mapped to screenshots, tickets, logs, or walkthrough notes later.

## 3) Assumptions
- The learning platform supports first-party email/password accounts, while MFA and SSO may be optional or environment-dependent.
- Some purchase and refund workflows may be self-serve in one environment and partner-scoped or manual-review in another; the commands remain valid as audit scenarios either way.
- Privacy-rights workflows such as export, correction, and delete are either implemented directly or routed through authenticated support workflows.
- Minor accounts require guardian consent, restricted communication scope, reduced community access, or a documented equivalent control.
- Instructor, support, trust-and-safety, enterprise-manager, and guardian privileges are distinct enough to support least-privilege auditing.
- Team and enterprise workflows assume seat management, scoped reporting, and commercial routing exist at least in a simulated QA path.
- Content moderation workflows assume policy acceptance, review states, takedown logging, and appeal visibility are in scope for audit even if operational ownership varies by environment.
- Notification controls distinguish essential transactional messaging from marketing or digest-style communications.
- Accessibility validation assumes the product can be exercised with keyboard-only navigation, screen readers, captions/transcripts, and reduced-motion or high-contrast preferences.
- Offline and low-bandwidth scenarios assume at least some learner-facing assets can degrade gracefully, retry, or provide downloadable alternatives.

## 5) User Commands
The command catalog below is grouped by role cluster. Every command includes preconditions, expected outcomes, a negative or edge variant, and audit focus areas.

### Learner Commands
- `L-01` Command: "As a guest, I want to preview Module 1, save it for later, and then create an account without losing the preview trail." Preconditions: guest preview exists; guest session can store lightweight wishlist or recently viewed items. Expected outcome: public and gated boundaries remain clear; signup explains whether guest state migrates; saved module context is preserved or explicitly disclosed as non-portable. Negative/edge variant: blocked or expired cookies disclose what will not carry over instead of silently losing state. Audit focus: `ux`, `privacy`, `security`.
- `L-02` Command: "I want to sign up on my phone, verify my email, and retry safely if the first link expires." Preconditions: email signup is enabled; verification emails can be captured in QA. Expected outcome: account starts as unverified with clear next steps; the first valid link activates the account; resend does not create duplicates. Negative/edge variant: expired or used links route the learner into a safe resend flow instead of a dead-end error page. Audit focus: `ux`, `security`, `reliability`.
- `L-03` Command: "I want to sign in from a new laptop, review the new-device warning, and turn on MFA for future logins." Preconditions: the account is active with prior device history; MFA enrollment is available if supported. Expected outcome: step-up checks complete before login succeeds; the new-device event is visible in history or notifications; MFA setup confirms without locking the learner out. Negative/edge variant: repeated bad codes or passwords are rate-limited without exposing account existence. Audit focus: `security`, `ux`.
- `L-04` Command: "I forgot my password, my first reset link expired, and I still need a safe recovery path without bypassing identity checks." Preconditions: password reset is enabled; a documented support-assisted recovery path exists or is explicitly unavailable. Expected outcome: expired links are rejected cleanly; new resets do not create conflicting recovery states; mailbox-loss scenarios route to the documented recovery channel. - `L-05` Command: "I want to compare plan details, apply a scholarship or pilot discount, and enroll in Intermediate Quantum Programming." Preconditions: self-serve or simulated billing is available; valid and invalid discount or scholarship codes exist in QA. ## 6) Risks & Mitigations
| Risk | Likelihood | Impact | Mitigation | Acceptance signal |
| --- | --- | --- | --- | --- |
| Fictional data accidentally resembles a real individual or account | Low | High | Restrict identifiers to synthetic names, `ql.test` or reserved domains, and `+1-555-010-xxxx` phone numbers only | Automated validation rejects non-synthetic email or phone patterns |
| Written artifact drifts from typed fixture source | Medium | Medium | Keep `apps/frontend/src/lib/audit-user-fixtures.ts` canonical and add a document smoke test tied to source counts and headings | Report headings and counts are asserted in CI |
| Role coverage is too narrow for audit realism | Low | High | Maintain guest, learner, creator, support, manager, and guardian coverage in one source set | Integration tests confirm all role clusters are present |
| Missing privacy-rights flows weaken audit readiness | Medium | High | Preserve export, delete, correction, consent, cookie, and notification controls across learner, support, and guardian scenarios | Commands cover data-rights workflows across multiple roles |
| Admin or support scenarios normalize over-permissioned access | Medium | High | Keep least-privilege assumptions explicit and include role-boundary checks in commands and review criteria | Admin commands verify redaction, scope limits, and audit logging |
| Minor-consent handling is underspecified | Medium | High | Include linked guardian-minor fixtures, pending-consent states, and revocation or restriction checks | Guardian and support commands validate approval before activation |
| Accessibility scenarios are underrepresented | Medium | High | Preserve accessibility-critical learner personas and multiple accessibility-focused commands | Validation confirms at least six accessibility or performance-sensitive commands |
| Enterprise governance scenarios omit procurement, SSO, or offboarding | Medium | Medium | Keep enterprise manager fixtures and commands for SSO, seats, billing, privacy reporting, and closure | Manager command cluster remains within required volume and category spread |
| Error handling coverage is too optimistic | Medium | Medium | Include expired links, unsupported browser messaging, rate limits, payment failures, upload drops, and degraded streaming | Error-path category remains covered across multiple role clusters |
| Weak traceability reduces audit evidence quality | Medium | Medium | Use stable account and command ids and require mapping to screenshots, logs, or tickets during execution | Every command references account ids and audit focus areas |

## 7) Acceptance Criteria
- The fixture package contains between `12` and `20` fictional accounts; the current canonical dataset contains `17`.
- The fixture package contains between `40` and `70` user commands; the current canonical dataset contains `45`.
- All required role clusters are represented: `guest`, `learner`, `instructor_creator`, `admin_support`, and `manager_guardian`.
- All eight journey categories are represented in the command library.
- `100%` of commands include command text, preconditions, expected outcome, negative variant, and audit-focus metadata.
- `100%` of referenced account ids in the command catalog resolve to defined fictional accounts.
- `100%` of email addresses use synthetic or reserved domains, and `100%` of phone numbers use the synthetic `+1-555-010-xxxx` pattern.
- Accessibility-sensitive scenarios appear in at least `6` commands.
- Privacy-heavy scenarios appear in at least `10` commands.
- The written artifact contains the exact eight required sections in the required order.
- The written artifact does not claim live-system observation, does not include real PII, and does not contain bypass, exploitation, or abuse instructions.

## 8) Test Strategy
**Manual walkthrough checklist**
- Validate guest preview to account creation, including cookie-choice and state-migration messaging.
- Validate first-time signup, email verification, MFA or new-device handling, password reset, and suspicious-login recovery.
- Validate learner purchase, scholarship, renewal-failure, invoice, certificate, progress, offline, and accessibility journeys.
- Validate creator activation, draft, upload, publish, schedule, accessibility metadata, revision, moderation, and collaborator scoping.
- Validate support workflows for duplicate-email handling, refunds, privacy-rights requests, deletion, moderation, rate-limit messaging, and security-disclosure routing.
- Validate manager and guardian workflows for SSO, seat governance, reporting scope, procurement documents, consent, minor progress visibility, and workspace offboarding.

**Automated checks**
- Keep `apps/frontend/tests/audit-user-fixtures.integration.test.ts` as the structural validation suite for counts, role coverage, journey coverage, edge flags, accessibility representation, privacy representation, and synthetic identifiers.
- Add a document smoke test so the Markdown artifact and source fixture package stay synchronized on required headings and current counts.
- Continue using existing frontend quality gates such as `npm run test:integration`; run `npm run build` whenever fixture-source or page-level code changes occur.
- If the public `/audit-fixtures` page is exercised, include accessibility checks such as axe or equivalent, plus Lighthouse accessibility and link validation where applicable.

**Security verification, non-intrusive**
- Verify auth-flow messaging is sanity-checked for password reset, MFA, suspicious-login, rate limiting, and unsupported-browser guidance without attempting to bypass controls.
- Verify support and admin scenarios expose redacted payment data, scoped permissions, and auditable action logs rather than raw secrets or over-broad data access.
- Verify TLS, security headers, and cookie attributes on any environment under test separately from this fixture package; those checks are complementary, not embedded in the fictional fixtures themselves.

**Data and privacy checks**
- Confirm consent capture and revocation flows are attributable to the correct guardian-minor relationship.
- Confirm export and delete workflows explain scope, retention, and session revocation clearly.
- Confirm notification-preference flows distinguish transactional, security, learning, and marketing communications.
- Confirm reporting views remain role-scoped and do not expose unrelated personal data.

**Traceability**
- Map each executed command id to one or more evidence artifacts such as screenshots, HAR captures, tickets, logs, or issue records.
- Record pass, fail, blocked, or follow-up-required status against each command id.
- Maintain the typed fixture library and this written artifact together so audit evidence always references the same synthetic user and command ids.
